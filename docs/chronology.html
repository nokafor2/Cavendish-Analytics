Phase 0 vs EKS — where are the servers?

There is no aws_instance in terraform/main.tf, but servers still exist — the role is split between an AWS-managed control plane and EC2 worker nodes created indirectly by EKS.

  • Phase 0: one EC2 you launched ran everything — the k3s control plane, Nginx Ingress, and your pod.
  • EKS control plane: the Kubernetes "brain" (API server, etcd, scheduler) is fully managed by AWS. You never see it as an EC2 — only the cluster_endpoint output.
  • EKS worker nodes: the eks_managed_node_groups block creates an Auto Scaling Group that launches the EC2 instances (2× t3.medium by default) where CoreDNS, kube-proxy, vpc-cni, the EBS CSI driver, and later your app pods actually run.

So EC2 is created — EKS owns its lifecycle instead of you declaring it. Confirm with kubectl get nodes (2 Ready nodes) or aws ec2 describe-instances --filters "Name=tag:eks:cluster-name,Values=cavendish-staging".

Subdomain DNS — automatic NS delegation

When domain_name is a subdomain of an existing Route 53 zone (e.g. eks.ayuadomain.com under ayuadomain.com), set parent_route53_zone_name in terraform/terraform.tfvars. terraform/main.tf then:

  1. Creates a new hosted zone for eks.yourdomain.com
  2. Looks up the parent zone (yourdomain.com) via data.aws_route53_zone.parent
  3. Writes an NS record in the parent zone pointing eks at the child zone's nameservers — no manual Console step

Your apex domain (e.g. ayuadomain.com → EC2) is untouched. These resources are not in Monday's targeted apply — they are created on a full terraform apply (Tuesday, when ACM + ExternalDNS need DNS). Verify after apply:

terraform output route53_zone_name_servers
nslookup -type=NS eks.ayuadomain.com
# Expected: NS records match the output nameservers
Troubleshooting — Terraform / EKS bootstrap
  • Error: NoSuchBucket on terraform init → run Step 1 bootstrap first; the backend bucket must exist before init
  • BucketAlreadyExists in bootstrap → S3 bucket names are globally unique; set a unique state_bucket_name in terraform/bootstrap/main.tf and the matching backend.bucket in terraform/main.tf
  • state lock held → a previous run crashed: terraform force-unlock <LOCK_ID> (only if you are sure no apply is running)
  • UnauthorizedOperation → the bootstrap IAM identity needs EC2, EKS, IAM, S3, ECR, and DynamoDB permissions
  • EKS node group stuck CREATE_FAILED → usually private subnets missing the kubernetes.io/role/internal-elb tag or no NAT egress; both are set by terraform/main.tf
  • ResourceInUseException on EKS access entry → the IAM user was already granted access manually (or a prior apply). Import into state instead of creating again: 
    # Add terraform/imports.tf with import blocks, then:
terraform apply
# Delete imports.tf after a successful apply
  • kubectl "couldn't get current server API group list" / provide credentials → run aws eks update-kubeconfig --name cavendish-staging --region eu-west-2 from your PC (no SSH). If kubeconfig is correct but auth still fails, your IAM user needs an EKS access entry — set enable_cluster_creator_admin_permissions = true in terraform/main.tf and terraform apply, or grant manually: 
    aws eks create-access-entry --cluster-name cavendish-staging \
  --principal-arn arn:aws:iam::ACCOUNT_ID:user/YOUR_USER --type STANDARD --region eu-west-2
aws eks associate-access-policy --cluster-name cavendish-staging \
  --principal-arn arn:aws:iam::ACCOUNT_ID:user/YOUR_USER \
  --policy-arn arn:aws:eks::aws:cluster-access-policy/AmazonEKSClusterAdminPolicy \
  --access-scope type=cluster --region eu-west-2
  • EntityAlreadyExists: Provider with url https://token.actions.githubusercontent.com → the GitHub OIDC provider already exists account-wide; terraform/irsa.tf uses a data source to reference it instead of creating a duplicate
  • unsupported Kubernetes version 1.29 → EKS drops old versions over time; set eks_cluster_version in terraform/variables.tf (default 1.32) and re-run terraform plan -out=tfplan
  • aws-ebs-csi-driver addon timeout in CREATING → the driver needs an IRSA role; it is installed via terraform/ebs-csi.tf with service_account_role_arn. Delete any stuck addon, then re-apply: 
    aws eks delete-addon --cluster-name cavendish-staging --addon-name aws-ebs-csi-driver --region eu-west-2
